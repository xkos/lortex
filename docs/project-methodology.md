# 项目从零到一方法论

> 适用于从想法阶段到可执行架构的全过程。项目启动时按需参考。
> 注意：此文件是参考文档，不是 IDE 规则。实现阶段的工作流已由 ai2ai-maintenance 和 git-workflow 规则覆盖。

## 总体流程

```
阶段一：需求分析（把想法变成结构化需求）
  ↓
阶段二：设计决策（逐个明确关键问题）
  ↓
阶段三：技术调研（为每个技术选型做调研对比）
  ↓
阶段四：UI 与交互设计（用户怎么用、页面怎么组织）
  ↓
阶段五：架构设计（分层、模块划分、接口边界）
  ↓
阶段六：文档体系与代码骨架（建立目录、AGENTS.md、功能域 PRD）
  ↓
阶段七：开始实现（见 IDE 规则：ai2ai-maintenance + git-workflow）
```

---

## 阶段一：需求分析

### 目标

将零散的想法、设想、痛点结构化为可执行的需求文档。

### 方法

1. **收集原始想法** — 让用户自由表达所有想法，不做过滤
2. **提炼核心矛盾** — 从所有想法中找到根本痛点，用一句话总结
3. **需求域划分** — 将所有想法按领域归类，形成域地图（通常 5-8 个域）
4. **逐域细化** — 每个域下列出具体需求项，编号管理
5. **优先级排序** — 按依赖关系和实用价值划分 Phase，每个 Phase 有明确目标
6. **提炼设计原则** — 从需求中归纳贯穿全局的设计理念（通常 4-6 条）

### 输出

`docs/prds/overview.md`，包含：项目定位、核心问题、域划分、各域需求、演进路线、设计原则

### 关键原则

- **先发散再收敛**：不要在收集阶段就否定想法
- **区分核心域和远期域**：核心域详细写，远期域记录方向即可
- **需求只说"做什么"**：不涉及"怎么做"（技术方案）

---

## 阶段二：设计决策

### 目标

逐个明确需求中的关键设计问题，形成明确的产品层面决策。

### 方法

1. **识别开放问题** — 从需求中提取所有不确定的设计问题
2. **逐个讨论** — 每个问题分析场景、列出选项、明确决策
3. **记录决策理由** — 不仅记录"选了什么"，还记录"为什么选"
4. **区分已定和待定** — 已明确的标记为决策，需要实践验证的标记为开放问题

### 输出

补充到 `docs/prds/overview.md` 的"关键设计决策"和"开放问题"章节。

### 关键原则

- **产品决策和技术决策分离**：对象模型、标签体系、导入流程等是产品决策（放 PRD）；数据库选型、框架选型等是技术决策（放 tech/）
- **不追求一次性完美**：能确定的先确定，不确定的标记为开放问题，留到实践中验证
- **决策要有场景支撑**：每个决策都应该有具体的使用场景来验证其合理性

---

## 阶段三：技术调研

### 目标

为每个需要技术选型的点做充分的调研对比，形成有理有据的决策。

### 方法

1. **明确调研问题** — 这个选型要解决什么问题，核心要求是什么
2. **列出候选方案** — 广泛搜索，不遗漏主流选项
3. **多维度对比** — 表格形式，维度包括：性能、成熟度、生态、学习成本、适用场景等
4. **深入分析 Top 候选** — 对最有可能的 2-3 个方案做详细的优劣分析
5. **做出决策** — 明确选择，列出理由，标注备选关注
6. **参考业内实践** — 调研大厂/知名项目在类似场景下的做法

### 输出

每个选型主题一个文档：`docs/tech/<主题>.md`

### 文档结构

```
1. 职责/目标
2. 核心要求
3. 候选对比（表格）
4. 各方案详细分析（优势、劣势）
5. 决策 + 理由
6. 备选关注
```

### 关键原则

- **调研结果和 PRD 分离**：技术调研放 docs/tech/，PRD 中只引用结论
- **标注时效性**：技术调研有时效，标注调研日期和版本号
- **备选关注**：技术在发展，标注未来可能的替代方案

---

## 阶段四：UI 与交互设计

### 目标

将需求转化为用户可感知的操作流程和界面结构。回答"用户怎么用"的问题。

### 前置条件

- 阶段一的需求已结构化（知道要做什么功能）
- 阶段二的设计决策已明确（知道关键的产品规则）
- 阶段三的技术选型已确定（知道 UI 框架的能力边界）

### 方法

1. **用户任务流分析** — 从 PRD 需求出发，梳理用户的核心操作路径
   - 列出所有用户任务，按频率分为高频、中频、低频
   - 每个任务画出操作步骤流（用户做什么 → 系统响应什么 → 下一步做什么）
   - 高频任务的路径要最短，低频任务可以藏深一些

2. **信息架构** — 页面/视图规划 + 导航结构
   - 确定主要视图（通常 4-7 个顶层视图）
   - 画出视图之间的导航关系（平级切换、层级跳转、弹窗覆盖）
   - 每个视图定义信息层级：主要内容区、辅助信息区、操作区

3. **交互模式决策** — 对关键交互点做设计选择
   - 每个决策列出可选方案、优劣对比、最终选择和理由
   - 典型决策点：列表 vs 网格、侧边面板 vs 弹窗、内联编辑 vs 独立页面、拖拽 vs 按钮操作

4. **竞品参考** — 借鉴同类产品的成熟 UI 模式
   - 找 2-3 个定位相近的产品，分析其 UI 的优点和不足
   - 不是抄，而是理解其设计背后的逻辑，取长避短

5. **状态与反馈设计** — 定义各种状态下的 UI 表现
   - 空状态（首次使用、无结果）
   - 加载状态（长时间操作的进度反馈）
   - 错误状态（操作失败、资源不可用）
   - 成功反馈（操作完成的确认）

6. **原型验证**（可选） — 用静态 HTML 原型确认布局和交互
   - 根据上述步骤产出的设计文档，生成关键页面的 HTML 原型
   - 目的是在实现前低成本确认布局、间距、信息层级是否合理
   - 确认后再用目标 UI 框架实现，原型作为参考保留
   - 具体生成规范见 rules/ 中的 ui-prototype 规则

### 输出

`docs/prds/ui-design.md`，包含：

```
1. 用户任务流（核心任务的操作步骤图）
2. 信息架构（视图列表 + 导航结构 + 各视图信息层级）
3. 交互模式决策（关键交互点的选择和理由）
4. 竞品参考分析
5. 状态与反馈设计
6. Phase 对应（每个 Phase 实现哪些视图和交互）
```

`ui-preview/`（可选），包含关键页面的 HTML 原型文件

### 关键原则

- **任务驱动，不是页面驱动**：先想清楚用户要完成什么任务，再决定需要什么页面
- **高频操作路径最短**：最常用的功能应该最容易触达，不超过 2 次点击
- **渐进式披露**：初始界面简洁，高级功能按需展开
- **状态可见**：用户始终知道系统在做什么、做完了没有、出了什么问题
- **UI 设计不依赖具体框架**：描述的是"用户看到什么、怎么操作"，不涉及具体组件实现

## 阶段五：架构设计

### 目标

基于需求和技术决策，设计整体软件架构。

### 方法

1. **确定分层** — 通常 3-4 层（UI → API → Service → Infrastructure）
2. **识别关键边界** — 哪些层之间需要可替换性？这些边界怎么设计？
3. **模块划分** — 核心业务 vs 可复用模块，依赖方向单向向下
4. **定义核心抽象** — 关键的 trait/interface，按能力拆细
5. **画数据流** — 核心场景的完整数据流（如导入流程、检索流程）
6. **Phase 对应** — 每个 Phase 激活哪些模块

### 输出

`docs/tech/architecture.md`

### 文档结构

```
1. 技术决策汇总（引用各选型文档）
2. 分层架构图 + 关键边界说明
3. 项目目录结构
4. 模块依赖关系
5. 核心 trait/interface 定义
6. 关键数据流
7. Phase 演进与模块对应
```

### 关键原则

- **核心业务和可复用模块分开放**：核心业务放主目录，可复用模块放独立目录
- **依赖单向向下**：可复用模块不依赖核心业务
- **渐进式抽象**：初期用具体类型，接口定义好备用，等需要时再泛型化
- **独立子项目自包含**：如果某个模块定位为未来可独立开源，它的文档应该放在自己目录下

## 阶段六：文档体系与代码骨架

### 目标

建立完整的文档体系和代码目录结构，为开发做好准备。

### 三类文档各司其职

| 文档类型 | 位置 | 受众 | 内容 |
|----------|------|------|------|
| PRD | `docs/prds/` 集中管理 | 人（产品/设计视角） | 做什么、为什么做 |
| 技术文档 | `docs/tech/` 集中管理 | 人（技术视角） | 怎么做、为什么这么做 |
| AGENTS.md | 跟着代码目录走 | AI 编码助手 | 开发这个模块要注意什么 |

### AGENTS.md 层级

```
项目根/AGENTS.md       — 极简路由，指向文档和代码
代码根/AGENTS.md       — 代码层级总览，模块关系
模块/AGENTS.md         — 职责、边界、依赖、规范、注意事项
```

### PRD 位置原则

- 产品功能的 PRD → `docs/prds/`（集中管理，方便全局浏览）
- 独立子项目的 PRD → 子项目自己的 `docs/prds/`（自包含，方便未来拆分）
- 纯工具库模块 → 不需要独立 PRD，AGENTS.md 足够

### 执行顺序

1. 建立代码目录结构（目录 + 空入口文件）
2. 编写根级和代码层级 AGENTS.md
3. 编写各模块 AGENTS.md
4. 从 overview.md 展开各功能域 PRD

---

## 文档体系

### 文档职责划分

| 类别 | 位置 | 维护者 | 内容 |
|------|------|--------|------|
| me2ai | `docs/prds/`, `docs/tech/`, `AGENTS.md`, rules/ | 人 | 需求、架构、编码规范 — "应该是什么样" |
| ai2ai | `docs/ai2ai/` | AI（人审核） | 项目状态、迭代记录、经验库 — "现在实际是什么样" |

## 迭代原则

- 首次迭代必须解决：框架搭建、环境问题、从开发到测试的流程
- 每次迭代 AI 必须提供可测试的内容
- 人的角色：定规则（me2ai）+ 审任务（task）+ 跑测试（iteration 中的可测试项）
- AI 的角色：写代码 + 维护 ai2ai + 执行已审核的 task
